## 구현 착수 순서

1. DB schema와 migration을 만든다.
2. 월드컵/후보 생성, 조회, 발행 API를 만든다.
3. seed 기반 bracket 생성과 play session API를 만든다.
4. 상세, 라운드 선택, 플레이, 결과 화면을 연결한다.
5. 홈/탐색/카테고리 유입 경로를 붙인다.
6. 만들기 3단계와 비밀번호 기반 수정을 붙인다.
7. 공유 이미지, 신고, 관리자 처리, 이벤트 로깅을 붙인다.
8. `qa-release-plan.md` 기준으로 모바일/데스크톱 회귀 테스트를 수행한다.

## MVP에서 흔들리면 안 되는 결정

- 기본 토너먼트 정책은 랜덤 샘플링이다. 전체 후보 부전승 모드는 설정 옵션으로 분리한다.
- 플레이 화면에는 광고나 방해 UI를 넣지 않는다.
- 비회원 생성은 허용하되, 수정은 비밀번호 해시 기반으로만 처리한다.
- 신고/관리자 처리는 P0에 포함한다. 댓글/좋아요/저장은 P1로 둔다.
- 공유 이미지는 P0에서 정사각형 1종을 우선 구현하고, 다른 비율은 여유가 있을 때 확장한다.

## 다음 단계 권장

구현에 들어가려면 먼저 프로젝트 스택을 확정하고 앱 scaffold를 만든 뒤, `architecture.md`의 도메인 모델을 기준으로 schema부터 시작한다. 이후 API와 프론트엔드 화면을 병렬로 붙이되, 플레이 세션과 match 선택 API가 가장 먼저 안정화되어야 한다.
